## Summary
Add a real SQL fixture integration layer that executes `tests/sql/smoke-load.sql` and compares results to `tests/golden/smoke-load.out`, while keeping dynamic function-surface smoke coverage for all implemented signatures. Expand `smoke-load.sql` to include comprehensive happy-path and edge-case checks across implemented methods, including expected-error scenarios.

- `crates/sqlite-presto-ext/tests/load_extension.rs` currently has:
  - extension load test
  - dynamic smoke test that calls all implemented `p_*` signatures with null-based invocation.
- SQL fixture folders (`tests/sql`, `tests/golden`) have README placeholders and no runner harness.
- Many implemented methods are currently unverified in SQL fixture/golden form.

## Research Findings
Skipped — high confidence in approach based on current test setup and SQLite CLI-driven integration patterns already used in the repository.

## Approach

### 1) Add SQL+golden integration harness
Implement a new integration test that:
- builds/loads the extension
- runs `tests/sql/smoke-load.sql` through `sqlite3`
- captures output deterministically
- compares output against `tests/golden/smoke-load.out`

Harness behavior:
- stable CLI mode (`.mode list`, `.headers off`, deterministic ordering in SQL)
- explicit failure context showing command stderr and diff location
- reusable helper functions for future SQL fixture growth.

### 2) Expand one big smoke fixture (`smoke-load.sql`)
Per your preference, keep one large SQL file and add sections for:
- numeric + bitwise
- text + regex
- binary + hash + encoding
- URL + IP + UUID
- overload coverage where applicable
- edge cases (null propagation, boundary values, invalid-input error paths where practical)

Each query will emit deterministic, diff-friendly rows (e.g., labels + normalized outputs), avoiding nondeterministic values (random/uuid raw values) or asserting stable derivatives (length/type/pattern checks).

### 3) Keep dynamic full-signature smoke as safety net
Retain and align the dynamic signature smoke test so all implemented SQL signatures continue to be exercised mechanically. This ensures broad method coverage even if fixture assertions are curated.

### 4) Tighten fixture documentation
Update `tests/sql/README.md` and `tests/golden/README.md` with:
- how fixture/golden comparison works
- expectations for adding new function tests
- output-format rules to keep goldens stable.

## Why This Approach
This gives two complementary layers:
- **comprehensive surface coverage** via dynamic signature smoke (all implemented methods/signatures)
- **behavioral edge-case regression protection** via deterministic SQL golden fixture.

It also preserves your requested single-file SQL layout while still being maintainable and CI-enforced.

## Alternatives Considered
- **Only dynamic smoke calls:** broad but too weak for semantic/edge regressions.
- **Many SQL files split by family:** cleaner scaling, but rejected per your preference for one big `smoke-load.sql`.
- **Rust-only assertion matrix without SQL fixtures:** stronger typing but misses the explicit SQL-fixture workflow you requested.

## Scope
- **In scope:**
  - integrate `tests/sql/smoke-load.sql` + `tests/golden/smoke-load.out` into cargo tests
  - expand smoke SQL to cover implemented methods and key edge cases
  - keep dynamic all-signature smoke coverage in integration tests
  - update fixture README docs
- **Out of scope:**
  - implementing currently stubbed SQL bridges (window/vector stubs)
  - changing function semantics/implementation logic
  - splitting fixtures into multiple SQL files

## Risks & Mitigations
| Risk | Impact | Mitigation |
|------|--------|------------|
| Single large SQL file becomes hard to navigate | Medium | Add strict section headers and stable label conventions |
| Golden output churn from nondeterministic functions | Medium | Assert deterministic derivatives (type/length/pattern), avoid raw random output checks |
| Expected-error output can vary | Medium | Use normalized assertions and stable error-trigger queries; keep matching deterministic |
| Duplicate coverage (dynamic + golden) increases runtime | Low | Keep dynamic smoke lightweight and fixture SQL deterministic |

## Dependencies
No new external dependencies expected. Uses existing `sqlite3` CLI-based integration pattern and Rust test tooling.
